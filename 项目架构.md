## 2. Chrome 扩展三层通信模型

Chrome MV3 扩展由三个相互隔离的执行上下文组成，通过 `chrome.runtime.sendMessage` 传递消息：

| 层 | 文件 | 权限 | 生命周期 |
|----|------|------|----------|
| **Popup**（弹窗） | `popup/App.tsx` | 仅限扩展 API | 用户开关弹窗时启动/销毁 |
| **Background**（后台） | `background/background.ts` + `agentOrchestrator.ts` | 全扩展 API + fetch | Service Worker，空闲时休眠 |
| **Content Script**（内容脚本） | `content_script/content.ts` + Adapters | DOM 访问 + 有限扩展 API | 随目标页面存活 |

**关键约束**：
- Content Script 能读写 DOM，但不能直接调用 fetch（避免暴露 API Key）
- Background SW 能调用 fetch，但无法访问任何页面的 DOM
- Popup 随时可能被销毁，不能承担任务执行责任
- SW 可能随时休眠，所有状态必须持久化到 `chrome.storage.local`

## 4. 各文件详解

### 4.1 `manifest.json`

扩展的配置文件，声明权限和入口点。

**权限说明**：

| 权限 | 用途 |
|------|------|
| `storage` | 读写 `chrome.storage.local`（TaskState、UserConfig）|
| `activeTab` | 获取当前活动标签页 ID |
| `scripting` | 向目标页面注入脚本（按需） |
| `tabs` | 导航标签页（`chrome.tabs.update`）、查询标签页信息 |
| `downloads` | 触发 CSV 文件下载 |

**host_permissions**（需要 Content Script 注入的域名）：
- `https://www.linkedin.com/*`
- `https://www.linkedin.cn/*`（中国区 LinkedIn InCareer）
- `https://www.indeed.com/*`
- `https://www.zhipin.com/*`（Boss直聘）
- `https://api.openai.com/*`、`https://api.deepseek.com/*`（LLM API fetch）

**Content Script 配置**：
- `run_at: document_idle`：DOM 加载完成后才注入，确保页面元素可查询

### 4.2 `background/background.ts`

**角色**：Service Worker 入口，消息路由器。

**核心职责**：接收所有 `chrome.runtime.sendMessage` 消息，分发给 `agentOrchestrator.ts` 处理。

**关键设计——消息分类处理**：

MV3 Service Worker 的消息通道有严格限制：如果不立即调用 `sendResponse` 或返回 `true`，通道会自动关闭。本文件将消息分为两类：

```typescript
// 火发即忘（content script 发来的结果通知）
const FIRE_AND_FORGET = new Set(['CONTENT_READY', 'LINKS_RESULT', 'JD_TEXT_RESULT'])
```

| 消息类型 | 处理方式 | 原因 |
|---------|---------|------|
| `CONTENT_READY`, `LINKS_RESULT`, `JD_TEXT_RESULT` | 立即 ACK（`return false`），异步处理 | Content Script 不需要返回值，快速释放通道 |
| `START_TASK`, `STOP_TASK`, `GET_STATE`, `EXPORT_CSV`, `RESET_DATA` | 保持通道（`return true`），等待异步结果后回复 | Popup 需要等待操作结果 |

还包含一个 `chrome.tabs.onUpdated` 监听器用于调试日志（记录标签页开始加载的时机）。

### 4.3 `background/agentOrchestrator.ts`（核心）

**角色**：Agent 的大脑，驱动整个自动化流程。

#### 平台配置（PLATFORMS 数组）

定义三个平台的完整配置：

```typescript
const PLATFORMS = [
  { name: 'LinkedIn', hostname, keywords: [...20个关键词], buildUrl: (keyword, page) => URL },
  { name: 'Indeed',   hostname, keywords: [...20个关键词], buildUrl: (keyword, page) => URL },
  { name: 'Boss直聘', hostname, keywords: [...24个关键词], buildUrl: (keyword, page) => URL },
]
```

每个平台包含：
- **keywords**：按 AI 方向分组的搜索关键词（AI/LLM、机器学习、NLP/CV、算法工程、MLOps）
- **buildUrl**：构建带翻页、时间过滤、经验过滤的搜索 URL

URL 过滤参数：

| 平台 | 时间过滤 | 经验过滤 |
|------|---------|---------|
| LinkedIn | `f_TPR=r7776000`（90天） | `f_E=1,2`（实习+入门） |
| Indeed | `fromage=90`（90天） | 无（依赖关键词） |
| Boss直聘 | `publishTime=3`（1个月） | `experience=101,102`（在校生+应届生） |

#### 核心函数

**`navigateTab(tabId, url)`**
导航前先重读 TaskState，若已停止则中止导航。并捕获 `chrome.tabs.update` 可能抛出的"No tab with id"错误（标签页被关闭时）。

**`onContentReady(tabId)`**
Content Script 加载完成时触发。
1. 验证 tabId 与 `state.activeTabId` 匹配
2. 检查状态是否为停止状态
3. 调用 `chrome.tabs.get` 获取当前 URL，检测是否为登录墙
4. 等待后**重读状态**（Stop 竞态保护）
5. 根据当前状态决定下一步：`searching` → 发 `EXTRACT_LINKS`；`visiting_detail` → 发 `EXTRACT_JD_TEXT`

**`onLinksResult(tabId, payload)`**
收到搜索结果页的链接列表时触发。
1. 过滤已访问 URL（dedup）
2. 统计连续重复数
3. **重读状态**（Stop 竞态保护）
4. 判断是否需要切换关键词（连续重复 ≥5 或达到最大翻页数）
5. 将新链接追加到 `pendingLinks`，导航到第一个详情页

**`onJdTextResult(tabId, payload)`**
收到岗位详情文本时触发。
1. 调用 `analyseJob`（LLM 分析，耗时最长）
2. **LLM 返回后重读状态**（Stop 竞态保护）
3. 若岗位符合条件：二次去重（title+company）→ 保存到 IndexedDB → 更新计数
4. 判断是否达到 50 条目标 → 触发 CSV 下载
5. 继续处理 `pendingLinks` 中的下一个 URL，或翻页

**`advanceToNextKeywordOrPlatform(state)`**
当前关键词耗尽时，切换到同平台下一个关键词；所有关键词耗尽时，切换到下一个已选平台。

**`skipCurrentPlatform(state, tabId)`**
检测到登录墙时直接跳过整个当前平台，导航到下一个平台的搜索页。

**`startTask(tabId, selectedPlatformIndices)`**
检测是否存在已有进度（`collected > 0` 或 `visitedUrls.length > 0`）：
- **有进度且未完成** → Resume：保留所有状态，仅更新 `activeTabId`，从 `pendingLinks[0]` 或当前搜索页继续
- **无进度或已完成** → Fresh Start：重置 TaskState，从第一个平台第一个关键词第一页开始

**`stopTask()`**
将 `status` 设为 `idle`，清除 badge。所有 async handler 会在下一个 await 点检测到 `idle` 状态后自动退出。

#### Stop 竞态保护机制

JS async/await 允许多个 handler 并发执行。STOP_TASK 可能在任意 await 点之间到达。保护方案：

```
onContentReady:  chrome.tabs.get(await) → 重读状态 → 若 idle 则 return
onLinksResult:   链接过滤计算 → 重读状态 → 若 idle 则 return
onJdTextResult:  analyseJob(await, 最长) → 重读状态 → 若 idle 则 return（但仍记录 visitedUrl）
```

### 4.4 `background/llmClient.ts`

**角色**：与外部 LLM API 通信，将原始 JD 文本转换为结构化 `JobData`。

#### 工作流程

```
原始文本（截断至5000字）
    ↓
buildUserPrompt() 构造 user message
    ↓
callLLMRaw() 调用 /chat/completions（最多3次重试，指数退避：1s/2s/4s）
    ↓
JSON.parse() 解析响应
    ↓
Zod safeParse() 校验结构
    ↓
检查 is_ai_engineer && is_campus_or_intern
    ↓
返回 JobData | null
```

#### System Prompt 关键规则

**`is_ai_engineer = true`** 的条件：AI Engineer、ML Engineer、LLM Engineer、Algorithm Engineer、Applied Scientist、NLP Engineer、CV Engineer、MLOps Engineer 等，主要职责是构建/训练/部署 AI 系统。

**`is_ai_engineer = false`** 的典型排除：
- 通用后端/前端/数据工程师（即使产品是 AI）
- AI 测试工程师 / QA
- AI 设计师 / 生图师
- DevOps / 运维（即使"AI 赋能"）
- AI 仅为辅助工具的岗位

**`is_campus_or_intern = true`** 宽松认定：intern、校招、应届、实习、0-2年经验、2025/2026届毕业生等均视为符合。

**LLM 参数**：`temperature: 0`（确定性输出）、`response_format: { type: 'json_object' }`（强制 JSON）。

### 4.5 `content_script/content.ts`

**角色**：Content Script 入口，按当前域名实例化对应的 Adapter，转发 Background 的命令。

#### 域名路由

```typescript
function createAdapter(): SiteAdapter | null {
  const host = window.location.hostname
  if (host.includes('linkedin')) return new LinkedInAdapter()  // 同时匹配 linkedin.com 和 linkedin.cn
  if (host.includes('indeed.com')) return new IndeedAdapter()
  if (host.includes('zhipin.com')) return new BossAdapter()
  return null
}
```

#### 消息处理

| 收到命令 | 操作 | 回复 Background |
|---------|------|----------------|
| `EXTRACT_LINKS` | 调用 `adapter.extractJobLinks()` | 发送 `LINKS_RESULT` |
| `EXTRACT_JD_TEXT` | 调用 `adapter.extractJobDetailText()` | 发送 `JD_TEXT_RESULT` |

**重要细节**：收到命令后立即同步调用 `sendResponse({ ok: true })` 关闭通道，再异步执行耗时的 DOM 操作。最终通过 `sendToBackground()` 将结果主动发回。

**页面加载完成通知**：Content Script 在 `document_idle`（DOM 完全加载后）注入，注入后立即发送 `CONTENT_READY`，触发 Background 的下一步决策。

### 4.6 `content_script/adapters/SiteAdapter.ts`

**角色**：Strategy 模式的接口定义，所有平台适配器必须实现此接口。

```typescript
export interface SiteAdapter {
  readonly platformName: string
  extractJobLinks(): Promise<string[]>          // 从搜索结果页提取岗位链接
  extractJobDetailText(): Promise<string>       // 从详情页提取原始文本
}
```

Background 只与此接口交互，不感知具体平台，实现开闭原则。

### 4.7 `content_script/adapters/LinkedInAdapter.ts`

**平台**：LinkedIn（linkedin.com + linkedin.cn InCareer）

**`extractJobLinks()`**：
1. `scrollToLoadCards()`：滚动 `.jobs-search-results-list` 容器触发懒加载
2. 查询 `a[href*="/jobs/view/"]` 和 `a[href*="/incareer/jobs/view/"]`
3. 从 pathname 提取数字 ID：`/jobs/view/(\d+)` 或 `/incareer/jobs/view/(\d+)`
4. 根据当前域名构造规范化 URL（保持在同一域名内，不跨域）

**`extractJobDetailText()`**：
1. `waitForJDLoad()`：等待骨架屏消失（最多6秒轮询）
2. `clickShowMore()`：点击"Show more"按钮展开完整 JD
3. 按优先级依次尝试多个 CSS 选择器提取标题、公司、地点、薪资、正文

**多域名支持**：同一 Adapter 同时支持 linkedin.com（VPN）和 linkedin.cn（InCareer，无需 VPN），通过 `hostname.includes('linkedin.cn')` 判断并构造对应规范 URL。

---

### 4.8 `content_script/adapters/IndeedAdapter.ts`

**平台**：Indeed（indeed.com）

**`extractJobLinks()`**：
1. 滚动页面触发懒加载
2. 查询多种选择器：`a.jcs-JobTitle`、`a[data-jk]`、`h2.jobTitle a` 等
3. 优先使用 `?jk=xxx` 参数构造规范化 URL（`/viewjob?jk=xxx`），去除追踪参数

**`extractJobDetailText()`**：
- 等待 800ms 确保页面渲染
- 使用 `[data-testid="..."]` 属性选择器提取结构化字段（对 DOM 变更更稳定）
- 主要目标元素：`#jobDescriptionText`

---

### 4.9 `content_script/adapters/BossAdapter.ts`

**平台**：Boss直聘（zhipin.com）

**`extractJobLinks()`**：
1. `waitForJobCards()`：轮询直到 `a[href*="/job_detail/"]` 出现（最多8秒），处理 SPA 慢渲染
2. `scrollToLoadCards()`：优先滚动 `.job-list-box` 等 **容器元素**（不是 window），触发容器内部懒加载
3. 使用宽泛选择器 `a[href*="/job_detail/"]` 提取所有岗位链接
4. 构造规范化 URL：`https://www.zhipin.com{pathname}`（去除查询参数）

**`extractJobDetailText()`**：
1. `waitForContentLoad()`：等待骨架屏（`.skeleton`, `.loading`）消失
2. 多选择器依次尝试：标题（`.job-name`）、公司（`.company-name`）、地点（`.location-address`）、薪资（`.salary`）
3. 提取 `.job-tags` 中的标签（经验/学历要求）
4. 主要正文：`.job-sec-text`，回退到 `.job-detail-box`

**Boss直聘特殊处理**：
- SPA 架构（Vue），路由切换不触发页面刷新 → 需要主动等待 DOM 渲染
- 职位列表有独立滚动容器，必须滚动容器而非 window
- 使用宽泛链接选择器而非依赖特定 class（class 名称频繁变化）

### 4.10 `shared/types.ts`

**角色**：全项目共享的 TypeScript 类型定义，是三个执行上下文的通信契约。

#### 核心类型

**`JobData`**：最终输出的岗位数据结构：
```typescript
interface JobData {
  title, company, location, salary,
  tech_tags: string[],
  requirements: string,
  source: string,   // "LinkedIn" | "Indeed" | "Boss直聘"
  job_url: string,
}
```

**`TaskState`**：Agent 完整状态，持久化到 `chrome.storage.local`：

| 字段 | 类型 | 说明 |
|------|------|------|
| `status` | `AgentStatus` | 当前状态（见状态机节） |
| `target` | number | 目标收集数量（默认50） |
| `collected` | number | 本次运行已收集数量 |
| `visitedUrls` | string[] | 已访问 URL 集合（去重） |
| `pendingLinks` | string[] | 待访问详情页 URL 队列 |
| `currentKeywordIndex` | number | 当前关键词在 keywords 数组中的位置 |
| `currentPage` | number | 当前搜索页码 |
| `pagesPerKeyword` | number | 当前关键词已消费页数 |
| `maxPagesPerKeyword` | number | 单关键词最大翻页数（默认10） |
| `consecutiveDupes` | number | 连续重复 URL 计数（触发关键词切换） |
| `currentPlatformIndex` | number | 当前平台在 PLATFORMS 中的索引 |
| `activeTabId` | number \| null | Agent 控制的标签页 ID |
| `savedJobKeys` | string[] | 已保存岗位的 `title\|\|company` 键（二次去重） |
| `selectedPlatformIndices` | number[] | 用户勾选的平台索引（0=LinkedIn, 1=Indeed, 2=Boss） |

**`Message`**（消息传递协议）：

```
Popup → Background:   START_TASK, STOP_TASK, GET_STATE, EXPORT_CSV, RESET_DATA
Background → Content: EXTRACT_LINKS, EXTRACT_JD_TEXT
Content → Background: CONTENT_READY, LINKS_RESULT, JD_TEXT_RESULT
```

### 4.11 `shared/storageUtils.ts`

**角色**：封装所有持久化操作，屏蔽底层存储细节。

#### `chrome.storage.local` 操作（TaskState + UserConfig）

| 函数 | 说明 |
|------|------|
| `getTaskState()` | 读取 TaskState，与 `DEFAULT_TASK_STATE` 合并（兼容字段新增） |
| `updateTaskState(partial)` | 读-改-写，返回更新后完整状态，自动更新 `lastUpdated` |
| `resetTaskState()` | 恢复为默认状态 |
| `getUserConfig()` / `setUserConfig()` | UserConfig 读写 |

**`getTaskState` 的合并设计**：存储中的旧数据可能缺少新增字段，合并 `DEFAULT_TASK_STATE` 保证向前兼容：
```typescript
return stored ? { ...DEFAULT_TASK_STATE, ...stored } : { ...DEFAULT_TASK_STATE }
```

#### IndexedDB 操作（JobData）

数据库：`agentic-job-scout`，版本1，Object Store：`jobs`

| 配置 | 值 |
|------|-----|
| keyPath | `job_url`（URL 作为主键，天然去重） |
| 索引 | `by_company`（支持按公司查询） |

| 函数 | 说明 |
|------|------|
| `saveJob(job)` | `db.put`（upsert，URL 重复时覆盖而非报错） |
| `getAllJobs()` | 获取所有岗位（用于 CSV 导出） |
| `getJobCount()` | 当前数据库岗位数 |
| `clearJobs()` | 清空整个 store（仅重置时调用） |

#### CSV 导出

`triggerCSVDownload()`：
1. `getAllJobs()` 从 IndexedDB 读取全部数据
2. 转换为 CSV 字符串（字段包含逗号/引号时用 `""` 转义）
3. 编码为 `data:text/csv;charset=utf-8,...` URL（MV3 Service Worker 不支持 `URL.createObjectURL`）
4. 调用 `chrome.downloads.download()` 触发浏览器下载

### 4.12 `popup/App.tsx`

**角色**：用户界面，提供配置输入、进度展示、操作控制。

#### 状态管理

```typescript
const [config, setConfig] = useState<UserConfig>(DEFAULT_USER_CONFIG)  // 用户配置
const [taskState, setTaskState] = useState<TaskState | null>(null)       // Agent 状态
const [selectedPlatforms, setSelectedPlatforms] = useState<number[]>([0,1,2]) // 选中平台
```

#### 轮询机制

Agent 运行时，每秒发送一次 `GET_STATE` 消息拉取最新进度（进度条和状态文字实时更新）。Agent 停止时自动取消轮询：

```typescript
useEffect(() => {
  if (isRunning) {
    pollRef.current = setInterval(() => chrome.runtime.sendMessage({ type: 'GET_STATE' }), 1000)
  }
}, [taskState?.status])
```

#### 关键操作函数

**`handleStart()`**：
1. 立即乐观更新 UI（状态设为 `searching`，Stop 按钮立即显示，无需等待 SW 响应）
2. `saveConfig()` 保存当前配置
3. `chrome.tabs.query({ active: true, lastFocusedWindow: true })` 获取活动标签页（用 `lastFocusedWindow` 避免返回 Popup 自身的 window）
4. 异步发送 `START_TASK`（不 await 响应，避免 SW 唤醒延迟导致界面卡顿）

**`handleStop()`**：发送 `STOP_TASK`，同时本地状态立即设为 `idle`（try-catch 防止 SW 休眠时发送失败）。

**`handleReset()`**：发送 `RESET_DATA`（Background 调用 `clearJobs()` + `resetTaskState()`），本地状态清空。

#### UI 逻辑

- **Start / Resume / Stop** 按钮根据状态动态切换：有未完成进度时显示"Resume"
- **`canStart`**：需同时满足：API Key 不为空 + Agent 未在运行 + 至少选择一个平台
- **CSV 按钮**：已收集 > 0 条时显示
- **重置按钮**：仅在 Agent 停止时显示

## 7. 数据存储方案

### 双存储架构

| | `chrome.storage.local` | IndexedDB（idb） |
|-|------------------------|-----------------|
| 存储内容 | TaskState、UserConfig | JobData 岗位列表 |
| 容量限制 | ~5MB | 无限制（磁盘空间） |
| 读写方式 | 异步键值对 | 异步事务型数据库 |
| 选择原因 | SW 睡眠/唤醒后可快速恢复状态 | 岗位数据量大，结构化，需要索引 |

### 为何不全用 `chrome.storage`

TaskState 中的 `visitedUrls`、`pendingLinks`、`savedJobKeys` 是数组，随运行时间增长可能很大。JobData 包含完整 JD 摘要文本。全部塞入 `chrome.storage` 容易超过 5MB 限制。

### 为何不全用 IndexedDB

IndexedDB 不支持直接序列化复杂状态并快速读写。`chrome.storage` 的读写延迟更低，适合频繁更新的 TaskState（每处理一个岗位都需要更新）。

## 8. 关键设计决策

### 1. Service Worker 无状态化
MV3 的 Service Worker 可能随时被 Chrome 休眠（通常空闲30秒后）。所有状态都持久化到 `chrome.storage.local`，SW 唤醒后从 storage 恢复，可无缝继续任务。

### 2. Content Script 只提取原始文本
各平台的 DOM 结构不稳定，频繁改版。Content Script 只做最简单的事——用 CSS 选择器抓取 `innerText`，不做任何语义解析。语义解析、字段提取、资格判断全部交给 LLM，Adapter 的实现因此更稳定、更容易维护。

### 3. Stop 竞态保护
JS 的 async/await 不是真正的互斥。STOP_TASK 消息可以在任何 await 点之间插入。每个 handler 在每次重要 await 之后都重读 TaskState，检测到 `idle` 立即退出，确保 Stop 指令能在1-2个 await 步骤内生效。

### 4. 断点续跑设计
`startTask()` 检测 `collected > 0 || visitedUrls.length > 0` 来判断是否有进度，有进度则 Resume（只更新 tabId），无进度则 Fresh Start（重置 TaskState）。JobData 数据库始终不在 Start 时清空，只在用户主动点「重置」时清空，允许多次 Stop/Start 累积收集。

### 5. 乐观 UI 更新
Popup 在发送 `START_TASK` 前立即更新本地状态（`searching`），无需等待 SW 响应。即使 SW 需要 1-2 秒唤醒，用户点击 Start Agent 后立刻看到 Stop 按钮，体验无延迟。

### 6. API Key 安全隔离
API Key 保存在 `chrome.storage.local`，只有 Background Service Worker 可以读取（通过 `getUserConfig()`）。Content Script 只接收命令和发送原始文本，永远不接触 API Key。即使内容脚本被注入的页面恶意读取，也无法获取 Key。

### 7. 多层去重策略
- **URL 级去重**：`visitedUrls` Set，防止重复访问同一详情页
- **内容级去重**：`savedJobKeys`（`title||company`），防止同一岗位在多城市发布时重复收录
- **数据库级去重**：IndexedDB 以 `job_url` 为 keyPath，`db.put` 是 upsert，即使重复写入也不会出错
